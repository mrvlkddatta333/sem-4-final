Abstract

Learning Management Systems are essential platforms in educational institutions for managing academic operations. As these systems grow in complexity, monolithic architectures become bottlenecks in scalability and maintainability.

This project focuses on redesigning three critical modules of an existing LMS at Codegnan IT Solutions Pvt Ltd, extracting them from the monolithic backend and redeveloping them as independent microservices:

📝 Exam Examiner Service

Exam creation, paper generation, WhatsApp notifications

📊 Exam Reports Service

Analytics, CQRS pattern, multi-role reporting

📄 Parent Reports Service

PDF generation, S3 upload, WhatsApp distribution

Built using Python Flask, MongoDB, Redis, Celery, and AWS S3. Migration followed the Strangler Fig Pattern for zero-downtime transition. Project period: December 2025 – March 2026.

Problem Statement & Objectives

⚠ Problems with Existing Monolith

  • Single deployment unit — 80+ endpoints in one Flask app
  • Shared resource pool — MongoDB pool exhausted during peak
  • No fault isolation — one crash takes down all modules
  • 90+ dependencies installed for every module
  • No independent scaling — entire app scales uniformly
Impact: During exam generation (500+ WhatsApp notifications), connection pool exhaustion caused timeouts for unrelated operations.

🎯 Project Objectives

  • Extract 3 compute-intensive modules as independent microservices
  • Implement clean layered architecture with separation of concerns
  • Apply Pydantic v2 validation and JWT-based RBAC
  • Enable independent deployment and targeted scaling
  • Achieve zero-downtime migration using Strangler Fig pattern
  • Implement async processing for long-running operations
  • Deploy each service with its own health monitoring

Layered Architecture & Design Patterns

API Layer
HTTP handling | Pydantic validation | JWT auth | Error responses
▼
Service Layer
Business logic | Orchestration | Domain rules | Workflows
▼
Repository Layer
MongoDB queries | Aggregation pipelines | Data mapping
▼
Integration Layer
WhatsApp API | AWS S3 | Redis cache | External services

Strict Layer Rules

  • API: No business logic, no DB access
  • Service: No HTTP handling, no direct DB queries
  • Repository: No business decisions — only data access
  • Integration: External clients with rate limiting

Design Patterns Applied

  • CQRS — Read/write separation (Reports)
  • Strategy — Exam type handling (Examiner)
  • Factory — Dynamic repo & document creation
  • Repository — Abstracted data access
  • Strangler Fig — Incremental decomposition

Future Enhancements & Lessons Learnt

🚀 Future Enhancements

  • API Gateway — Kong / AWS API Gateway for single entry point
  • Database-per-Service — Complete data isolation
  • Kubernetes — Container orchestration and auto-scaling
  • CI/CD Pipeline — GitHub Actions / AWS CodePipeline
  • Distributed Tracing — OpenTelemetry for monitoring
  • Event-Driven Architecture — RabbitMQ / AWS SQS
  • Analytics Dashboard — Prometheus + Grafana
  • Blue-Green Deployment — Zero-downtime updates

📚 Key Lessons Learnt

  • Strangler Fig enables zero-downtime migration — commenting out monolith routes after deployment
  • Layered architecture made codebase significantly easier to understand vs monolith's flat structure
  • Pydantic validation at API boundary catches most invalid inputs before business logic
  • Redis caching for enum mappings dramatically improved response times
  • Celery + Redis provided robust async processing for PDF generation
  • CQRS pattern simplified the Reports Service by focusing only on read operations
Limitations: Shared MongoDB cluster | 3 of 15+ modules migrated | No API Gateway | No CI/CD | No load testing
